# 客户演示话术

## 1. 开场 30 秒

我们这个阶段不是一上来就做洗车机硬件，而是先把无人洗车里最关键的识别、策略、空间、路径、覆盖和展示闭环做成软件仿真原型。这样在真正接机械、喷嘴、PLC 之前，先把逻辑跑通，把风险提前暴露出来。

今天演示的重点不是说它已经能控制真实设备，而是证明：从一张车辆图片开始，系统可以识别车型，选择洗车策略，生成空间模型、喷嘴覆盖、流程状态机和抽象路径，并把结果用客户能看懂的方式展示出来。

## 2. 展示 Stage1

第一步是车辆识别。我们输入一张车辆图片，系统会先做车辆检测，再裁切车辆区域，然后用三分类模型判断它是 `sedan`、`suv` 还是 `mpv`。

识别结果会写入一个标准文件：

```text
vehicle_type_lab\outputs\predictions\vehicle_type_result.json
```

这个 JSON 是后面仿真系统的入口。也就是说，AI 识别结果不是停留在页面上，而是能继续驱动后面的洗车策略。

## 3. 展示 Stage2

第二步是洗车仿真主链路。系统根据车型结果加载车辆尺寸模型和 `wash_profile`，然后生成洗车策略、洗车房空间模型、喷嘴覆盖计划、流程状态机、抽象路径点和覆盖率报告。

这一阶段的价值是：洗车逻辑先在软件中变得可检查、可解释。比如这个车型该用什么策略，哪些区域需要重点覆盖，流程状态如何切换，覆盖率是否达到目标，都可以先在软件里看到。

## 4. 展示 Stage3

第三步是可视化展示。我们把阶段2生成的 JSON 结果整理成 2D 可视化、时间轴动画和客户展示页。

2D页面更适合看空间、车辆包络、喷嘴路径和覆盖区域。时间轴动画更适合看洗车流程从预冲洗、泡沫、清洗、轮毂到风干的状态变化。客户展示页更适合汇报项目价值、当前能力和下一步路线。

## 5. 强调当前价值

当前最重要的价值是：我们已经把一个概念变成了可运行的软件原型。它可以帮助团队在硬件投入之前，先验证流程逻辑、喷嘴区域、路径点和覆盖率估算，降低直接上机械和 PLC 的试错成本。

## 6. 强调边界

也要说明清楚：当前还不是硬件控制系统，不代表真实清洗效果，也没有做流体仿真或真实机械轨迹规划。现在的覆盖率是基于抽象路径和配置参数的估算，不是实车洗净率。

## 7. 下一步

如果继续推进，下一步可以进入真实路径规划和运动约束，把现在的抽象路径逐步变成更接近机械运动的路径。再往后可以进入 PLC/硬件联调，最后再扩展到商业化后台、设备管理和现场数据闭环。
